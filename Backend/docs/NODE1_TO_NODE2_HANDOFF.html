Node 1 공유 및 연계 가이드

Node 1 자료를 처음 공유하는 기준으로, 프론트 API 응답과 LangGraph 내부 state를 분리해서 정리했습니다.

0. 이번 수정 요약

지훈님 피드백을 반영해 프론트 출력용 데이터와 LangGraph state용 데이터를 분리했습니다. 기존 프론트 API는 상세 응답을 유지하고, LangGraph에는 노드 연계에 필요한 최소 판정값만 넘깁니다.
구분 수정 전 수정 후
프론트 API 상세 카드, 이유, 근거, 추가 질문을 응답 그대로 유지. 화면 표시와 사용자 안내에 사용
LangGraph state 프론트 표시용 카드, profile, source_refs까지 함께 포함 공통 state에는 profile을 보관하고, Node 1 patch에는 후보 공급유형, 검토 필요 여부, 누락 key, 탈락 사유만 전달
추천 공급유형 Node 1 state에 recommended_supply 자리 포함 Node 2 책임으로 분리. Node 1은 후보군만 전달
profile Node 1 adapter가 다시 복사해서 전달 graph 시작 시점에 공통 state로 추가. Node 1 patch에서는 다시 복사하지 않음

1. 전체 역할

노드 담당 현재 기준
Node 1 준억 사용자 프로필 기반으로 신청 후보 공급유형을 rule 기반 판별. RAG는 근거 출처 보강에만 사용
Node 2 은진 Node 1 후보군을 받아 가점 계산, 비교, 브리핑, 추천 공급유형 결정
Node 3~4 동윤 공고 조건과 전략 흐름 처리
Node 5 지훈 앞선 노드 결과와 공고 정보를 받아 최종 전략/리포트 흐름 처리

2. 팀원별 확인 포인트

은진님

프론트 연동은 기존 POST /api/v1/diagnose/profile 응답을 보면 됩니다. Node 2 연계는 build_node1_graph_state()의 최소 state를 기준으로 보면 됩니다.

동윤님

Node 1은 공고 정보나 전략 생성까지 판단하지 않습니다. 이후 graph 공통 state에 profile, announcement, Node 2 결과가 채워지는 흐름을 가정합니다.

지훈님

Node 5에서 필요한 값은 Node 2 이후에 정리되는 것을 전제로 합니다. Node 1은 recommended_supply를 결정하지 않고 후보 신호만 남깁니다.

공통

화면 표시용 응답과 graph 내부 state는 목적이 다릅니다. API 응답은 풍부하게, graph state는 작고 명확하게 유지합니다.

3. 현재 Node 1 범위

현재 Node 1에서 다루는 공급유형은 다음입니다.

  • 일반공급
  • 신혼부부 특별공급
  • 다자녀 특별공급
  • 생애최초 특별공급
  • 신생아 특별공급
  • 노부모부양 특별공급
  • 기관추천 특별공급
기관추천 특별공급은 프론트 API 카드에는 남겨두되, 현재 Node 2 연계용 graph state에서는 우선 제외했습니다. 별도 추천/증빙 흐름이 필요해서 일반적인 가점 계산 흐름과 다르기 때문입니다.
노부모부양 특별공급의 핵심 조건인 만 65세 이상 직계존속 3년 이상 부양 여부는 현재 기본 profile 입력에 포함되어 있지 않습니다. Node 1은 무주택 세대주 조건을 먼저 보고 후보 가능성이 있으면 elderly_support를 node1_missing_profile_keys에 담아 후속 확인 항목으로 넘깁니다.

4. 프론트 기본 입력 여부

현재 기준에서는 모든 사용자에게 노부모부양 여부를 처음부터 묻지 않는 흐름을 권장합니다.

항목 현재 처리 이유
elderly_support 기본 profile에는 없음. Node 1에서 후보 가능성이 있으면 후속 질문 key로 전달 노부모부양 특공 후보가 아닌 사용자에게는 불필요한 질문이 될 수 있어 입력 부담을 줄이기 위함
profile.is_household_head 기본 profile에 포함 노부모부양뿐 아니라 여러 공급유형 판단에 영향을 줄 수 있는 기본 조건
profile.num_household_members 기본 profile에 포함 세대 구성 판단에 필요하고, 세대주 입력과 함께 확인하는 값
팀에서 노부모부양 여부를 첫 화면 기본 질문으로 받기로 결정하면 backend/src/schemas/diagnose.py의 UserProfile과 프론트 payload를 함께 수정해야 합니다. 지금 구조에서는 후속 질문 방식으로도 Node 2 연계가 가능합니다.

5. 확인할 파일

파일 확인 포인트
backend/src/services/rule_engine.py Node 1 자격 후보 판단의 중심 파일입니다. build_node1_profile_diagnoses()와 공급유형별 diagnose_*_by_profile() 함수를 확인하면 됩니다.
backend/src/services/node1_state_adapter.py Node 1 API 결과를 LangGraph 내부 state patch로 줄여주는 파일입니다. Node 2와 이후 노드 연계 시 핵심 확인 대상입니다.
backend/src/services/source_adapter.py RAG 검색 결과를 프론트 응답의 source_refs에 붙입니다. 자격 판단 자체는 rule engine이 담당합니다.
backend/src/schemas/diagnose.py /api/v1/diagnose/profile 요청/응답 스키마입니다.
backend/src/routers/diagnose.py FastAPI endpoint 위치입니다.

7. LangGraph State

LangGraph 내부 연계에는 초기 state와 Node 1 patch를 분리해서 사용합니다.

from src.services.node1_state_adapter import (
    build_initial_graph_state,
    build_node1_graph_state,
)

state = build_initial_graph_state(request)
state.update(build_node1_graph_state(request))

초기 state는 프론트 입력에서 온 공통 profile을 담습니다.

필드 의미
node1_available_supply_types Node 2가 비교 대상으로 삼을 수 있는 후보 공급유형 enum 목록
node1_supply_checks 공급유형별 후보 여부, 검토 필요 여부, 누락 key, 검토 사유, 탈락 사유
node1_missing_profile_keys Node 2 또는 프론트가 추가 입력을 요구할 때 참고할 key 목록
node1_need_review Node 1 후보군 중 추가 확인이 필요한 항목이 있는지 여부
함수 역할
build_initial_graph_state() 프론트 입력을 graph 공통 state로 정규화합니다. 여기에는 profile이 포함됩니다.
build_node1_graph_state() Node 1 판별 결과만 state patch로 반환합니다. 여기에는 profile을 다시 넣지 않습니다.
status 의미
CANDIDATE 현재 입력값 기준으로 후보로 볼 수 있고, Node 2에서 비교 대상으로 사용할 수 있음
NEED_REVIEW 후보 가능성은 있으나 공고 조건 또는 추가 입력 확인이 필요함
NOT_APPLICABLE 현재 입력값 기준으로 해당 공급유형 가능성이 낮아 Node 2 비교 후보에서 제외

8. Node 2에서 채울 값

값 담당 설명
recommended_supply Node 2 Node 1 후보 중 가장 유리한 공급유형. Node 1에서는 결정하지 않습니다.
ranked_supply_cards Node 2 후보 공급유형별 순위, 점수 요약, 브리핑 카드
score_results_by_supply_type Node 2 가점 계산기 또는 MCP tool 결과
briefing_message Node 2 사용자에게 보여줄 1차 브리핑 문장

9. 테스트 방법

  1. 백엔드 실행: uvicorn src.main:app --reload
  2. 프론트 실행: streamlit run frontend/streamlit_app.py
  3. 프론트에서 자가진단 입력 후 결과 카드 확인
  4. 프론트 응답에서는 candidate_supply_types, next_questions, source_refs 확인
  5. Node 2 연계에서는 build_node1_graph_state() 결과의 node1_available_supply_types와 node1_supply_checks 확인
