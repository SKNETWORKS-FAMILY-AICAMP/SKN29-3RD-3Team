# 청약제도안내 마크다운(MD) 청킹 전략

## 1. 문서 개요

| 항목 | 내용 |
|------|------|
| 입력 파일 | 청약제도안내.md |
| 형식 | Markdown (10개 청약Home 크롤링 문서를 합친 통합 파일) |
| 분량 | 1,135줄 / H1 10개, H2 44개, H3 32개 |
| 출력 | langchain `Document` 리스트 → ChromaDB `guide_chunks` |
| 최종 청크 수 | 72개 |

다른 5개 collection(법령/FAQ/매뉴얼/LH가이드/web_faq)과 달리, 이 문서는 **표 형태 데이터(소득기준, 가점표, 배점표 등)의 비중이 매우 높다**는 특징이 있습니다. 일반적인 글자수 기반 분할(`RecursiveCharacterTextSplitter`)을 그대로 쓰면 표가 청크 경계에서 잘리는 문제가 발생하여, 표를 보존하는 전용 청커를 만들었습니다.

---

## 2. 핵심 문제: 표가 잘리는 현상

기존(팀원 작성) 코드는 `chunk_size=1200`으로 단순 분할했는데, 다음과 같이 표 중간에서 끊기는 문제가 있었습니다.

```
[청크 경계 1200자 지점]
| 4년 이상 ~ 5년 미만 | 10 | 12년 이상 ~ 13년 미만 | 26 |
─────────────────────── (여기서 잘림) ───────────────────────
| 5년 이상 ~ 6년 미만 | 12 | 13년 이상 ~ 14년 미만 | ...
```

→ "5년 이상 6년 미만은 몇 점?" 질문 시, 표 헤더("무주택기간 (가점상한: 32점)")가 없는 청크가 검색되어 맥락을 잃습니다.

이를 해결하기 위해 **4단계 청킹 전략**을 설계했습니다.

---

## 3. 청킹 전략 (4단계)

### 1단계: H1/H2/H3 헤더 기준 1차 분할

`MarkdownHeaderTextSplitter`로 마크다운 헤더(`#`, `##`, `###`) 단위로 1차 분할합니다. 이때 각 청크에 `Header_1/2/3` 메타데이터가 자동 부여됩니다.

```python
headers_to_split_on = [
    ("#", "Header_1"),
    ("##", "Header_2"),
    ("###", "Header_3"),
]
```

H1은 원본 10개 문서 중 어느 것에서 왔는지를 나타내며, `doc_title` 메타데이터로도 별도 저장합니다.

### 2단계: 1,200자 이하 → 그대로 청크화

분할된 섹션이 `MAX_CHUNK_SIZE`(1,200자) 이하면 추가 분할 없이 그대로 사용합니다. (표가 있더라도 한 청크 안에 통째로 들어가므로 잘릴 위험이 없음)

### 3단계: 1,200자 초과 → 표 인식 분할

섹션이 1,200자를 넘으면 `split_into_text_and_tables()`로 텍스트와 표(`|...|` 블록)를 순서대로 분리한 뒤, `pack_pieces()`가 다음 원칙으로 재조합합니다.

- **표는 절대 쪼개지 않는다** (단, 4단계의 거대 표는 예외)
- 표 직전의 설명 텍스트(예: `### 가점항목 및 점수`, `**무주택기간**`)는 가능한 한 표와 같은 청크에 포함
- 의미 없는 조각(`---` 구분선만 남은 텍스트)은 제거

```
[원본 섹션 - 2,165자]
### 가점항목 및 점수 (총점: 84점)
**무주택기간 (가점상한: 32점)**
| 무주택기간 | 점수 | ... |
**부양가족 수 (가점상한: 35점)**
| 부양가족 수 | 점수 |
**입주자저축 가입기간 (가점상한: 17점)**
| 가입기간 | 점수 |

        ↓ pack_pieces

[청크 - 1,209자] (3개 표 + 헤더 텍스트가 한 덩어리로, 표 잘림 없음)
```

### 4단계: 거대 표(>1,200자) → 행 단위 동적 분할

표 하나가 1,200자를 넘는 경우(예: "공공주택 소득기준" 표, 원본 4,921자), `split_large_table()`이 **헤더 행(컬럼명 + 구분선)을 모든 조각에 복제**하면서 데이터 행을 글자 수 기준으로 묶어 분할합니다.

```
[원본 표 - 4,921자, 16개 데이터 행]
| 공급유형 | 공급방식 | 소득구분 | 기준 | 3인이하 | 4인 | 5인 |
|---------|---------|---------|------|--------|-----|-----|
| 다자녀가구... | (16행) ...

        ↓ split_large_table (max_size=1200)

[조각1 - 1,128자] 헤더행 + 데이터 6행
[조각2 - 1,121자] 헤더행 + 데이터 6행
[조각3 -  628자] 헤더행 + 데이터 4행
```

각 조각이 독립적으로도 "이게 무슨 표인지" 알 수 있도록 헤더 행을 유지하는 것이 핵심입니다. 분할된 거대 표 조각들은 서로 합쳐지지 않고 각각 독립 청크가 됩니다.

---

## 4. 표 → 자연어 문장 변환 ("표 설명" 추가)

표 형태(`| 항목 | 내용 |` 등)는 OpenAI 임베딩이 자연어 질문과의 의미적 유사도를 잘 잡지 못하는 문제가 있었습니다(검색 거리 0.95~1.0대). 이를 개선하기 위해 모든 표 청크에 **원본 표 + 자연어 변환 문장**을 함께 담습니다.

### 변환 규칙 (`table_to_sentences`)

표의 컬럼 구조에 따라 3가지 패턴으로 자동 분기합니다.

**① 2컬럼 항목/내용형** (헤더가 `항목`, `구분`, `용어`, `지역`, `주거전용면적` 등)

```
입력:
| 항목 | 내용 |
|------|------|
| 대상주택 | 전용면적 85㎡ 이하의 분양주택 |
| 소득기준 | 월평균소득 130% 이하 |

출력:
대상주택: 전용면적 85㎡ 이하의 분양주택
소득기준: 월평균소득 130% 이하
```

**② 컬럼명 반복형** (예: `무주택기간 | 점수 | 무주택기간 | 점수`처럼 한 행에 두 항목 쌍이 있는 표)

```
입력:
| 무주택기간 | 점수 | 무주택기간 | 점수 |
|----------|------|----------|------|
| 1년 미만 | 2 | 8년 이상~9년 미만 | 18 |

출력:
무주택기간: 1년 미만, 점수: 2
무주택기간: 8년 이상~9년 미만, 점수: 18
```

**③ 일반 다컬럼형** (그 외 모든 표)

```
입력:
| 공급유형 | 공급방식 | 소득구분 | 기준 | 3인이하 |
|---------|---------|---------|------|--------|
| 다자녀가구 | 우선공급(90%) | 부부 중 한명 소득 | 120%이하 | ~9,040,516 |

출력:
공급유형: 다자녀가구, 공급방식: 우선공급(90%), 소득구분: 부부 중 한명 소득, 기준: 120%이하, 3인이하: ~9,040,516
```

### 최종 청크 형태

원본 표는 LLM 답변 생성 시 정확한 형태로 참고할 수 있도록 그대로 유지하고, 그 아래에 `표 설명:` 섹션으로 자연어 문장을 추가합니다.

```
| 항목 | 내용 |
|------|------|
| 대상주택 | 전용면적 85m² 이하의 분양주택 |
| 소득기준 | ... 130% 이하 |

표 설명:
대상주택: 전용면적 85m² 이하의 분양주택
소득기준: ... 130% 이하
```

## 5. 헤더 컨텍스트 prefix

`MarkdownHeaderTextSplitter`는 헤더를 metadata로만 분리하고 본문에서 제거하기 때문에, 표만 남은 청크는 본문만 봐서는 "무슨 주제인지" 알 수 없는 문제가 있었습니다. 이를 위해 모든 청크 본문 맨 앞에 헤더 경로를 prefix로 삽입합니다.

```python
def build_prefix(meta):
    parts = [meta.get(f"Header_{i}") for i in (1,2,3)]
    return "[" + " > ".join(parts) + "]\n\n"
```

```
[청약제도안내 - 특별공급 > 생애최초 주택구입 특별공급 > 생애최초 (공공주택)]

| 항목 | 내용 |
|------|------|
| 대상주택 | 전용면적 85m² 이하의 분양주택 |
...
```

이 prefix 덕분에 "생애최초 특별공급 자격" 같은 질문에서 검색 거리가 1.0 → 0.7대로 개선되었습니다.

---

## 6. Metadata 설계

```python
metadata = {
    "Header_1":    "청약제도안내 - 특별공급",          # 원본 .md 파일 단위
    "Header_2":    "생애최초 주택구입 특별공급",
    "Header_3":    "생애최초 (공공주택)",
    "source":      "청약Home 청약제도안내",
    "source_year": "2026",
    "doc_title":   "청약제도안내 - 특별공급",          # = Header_1
    "chunk_level": "section",                          # 또는 "table"
}
```

| chunk_level | 의미 |
|---|---|
| `section` | 표가 없거나, 1,200자 이내라 분할이 필요 없었던 일반 섹션 |
| `table` | 표가 포함되어 표 인식 분할/자연어 변환이 적용된 청크 |

---

## 7. 노이즈 제거

마크다운 구분선(`---`)이 단독으로 남는 의미 없는 청크를 두 단계에서 제거합니다.

1. `pack_pieces` 1단계: `text` 타입 조각이 `-`, `|`, `:`, 공백으로만 구성되면 버림
2. 각 청크 생성 직후: `re.sub(r'\n+-{3,}\s*$', '', ...)`로 청크 끝의 `---` 제거

---

## 8. 처리 결과 요약

| 항목 | Before (단순 1200자 분할) | After (현재 청커) |
|------|------|------|
| 표 잘림 | 발생 (예: 무주택기간 표 중간 절단) | **0건** |
| 1,200자 초과 청크 | 9개 섹션 | 1개 (1,439자, 표 자체 크기) |
| 의미 없는 빈 청크 | - | 0개 |
| metadata | `Header_1/2/3`만 | + `source`, `source_year`, `doc_title`, `chunk_level` |
| 검색 거리 (생애최초 표 예시) | 0.95~1.0 | 0.69~0.72 |
| 최종 청크 수 | - | 72개 |

## 9. 전체 처리 흐름

```
청약제도안내.md (1,135줄)
        ↓
MarkdownHeaderTextSplitter (H1/H2/H3 분할)
        ↓
각 섹션 길이 확인
   ├─ ≤1,200자 ─────────────────────────────┐
   │                                          │
   └─ >1,200자                                │
        ↓                                     │
   split_into_text_and_tables (텍스트/표 분리) │
        ↓                                     │
   pack_pieces                                │
        ├─ 표 >1,200자 → split_large_table    │
        │   (헤더행 유지, 행 단위 동적 분할)    │
        └─ 텍스트+표 묶음 재구성                │
        ↓                                     │
        └─────────────────┬───────────────────┘
                           ↓
              augment_with_table_description
              (표 청크에 "표 설명:" 자연어 문장 추가)
                           ↓
                     build_prefix
              ("[H1 > H2 > H3]" 헤더 경로 삽입)
                           ↓
                    72개 최종 청크
                           ↓
                  ChromaDB (guide_chunks)
```
