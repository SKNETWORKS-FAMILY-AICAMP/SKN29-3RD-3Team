# 청약 법령 RAG 문서 청킹 전략 (완성본)

## 목표

일반적인 `CharacterTextSplitter`처럼 단순히 글자 수 기준으로 자르지 않는다.
법령이 가진 구조(장 → 조 → 항 → 호 → 목)를 최대한 보존하여 검색 정확도와 근거 추적성을 높인다.

---

## 1. 법령 계층 구조

```
장(章)  : 제1장 총칙 / 제2장 입주자저축 / ...  ← metadata로만 활용, 청킹 단위 아님
 └─ 조(條)
     ├─ 항(項)      : ① ② ③ ...
     │   ├─ 호(號)  : 1. 2. 3. ...
     │   │   └─ 목(目) : 가. 나. 다. ...
     └─ 호(號)      : 항 없이 바로 호가 오는 경우 (제2조 등, 범용 규칙)
         └─ 목(目)  : 가. 나. 다. ...
```

---

## 2. 전처리

청킹 전에 다음 세 가지를 처리한다.

### 2-1. 부칙 제거

부칙은 시행일·경과규정으로 RAG 검색 대상 가치가 없으므로 **자동으로 제거**한다.

```python
# "부칙" 이후 텍스트 전부 제거
m = re.search(r"(?m)^부칙\b", text)
if m:
    text = text[:m.start()]
```

### 2-2. 장(章) 구분 정보 수집

장 자체는 청킹 단위가 아니지만, **각 조가 어느 장에 속하는지** 미리 매핑해두어 metadata에 포함시킨다.

```
제1장 총칙                → 제1조 ~ 제4조
제2장 입주자저축           → 제5조 ~ 제14조의2
제3장 입주자모집 및 주택공급 신청 → 제15조 ~ 제24조의7
제4장 주택공급 방법        → 제25조 ~ 제49조
제5장 입주자 선정 및 관리  → 제50조 ~ 제61조
제6장 보칙                → 제62조 ~ 제63조
```

나중에 장 단위 필터 검색이 가능해진다.

```python
# 예: 특별공급 관련 조항만 검색
collection.query(
    query_texts=["신혼부부 소득 요건"],
    where={"chapter": "제4장 주택공급 방법"},
    n_results=5,
)
```

### 2-3. 조제목 + 항기호 분리

HWP 원문에서 조제목과 첫 번째 항이 같은 줄에 붙어 있는 경우가 있다.
항 패턴이 줄 시작에서만 매칭되도록 **줄바꿈을 삽입**한다.

```
# 원문
제4조(주택의 공급대상) ① 주택의 공급대상은 ...

# 전처리 후
제4조(주택의 공급대상)
① 주택의 공급대상은 ...
```

---

## 3. 청킹 흐름

```
전처리 (부칙 제거 → 장 매핑 → 조제목/항 분리 → 개정이력 제거)
        ↓
모든 법령을 조 단위로 분리
        ↓
조 길이 확인
        ↓
  ┌──────────────────────────────────────────┐
  │ article_length <= ARTICLE_THRESHOLD      │
  │ → 조 전체를 하나의 청크로 저장            │
  └──────────────────────────────────────────┘
        ↓ (초과 시)
  항이 존재하는가?
  ├─ YES → 항 단위로 분리
  └─ NO  → 호 단위로 분리  ← 제2조 등 범용 규칙
        ↓
  각 항/호 길이 확인 (병합 포함)
  ├─ 누적 길이 <= PARAGRAPH_THRESHOLD → 항 병합하여 하나의 청크
  ├─ 항 초과 → 호 단위로 분리
  │       ↓
  │   누적 길이 <= ITEM_THRESHOLD → 호 병합하여 하나의 청크
  │   호 초과 → 목 단위로 분리
  │           ↓
  │       목도 초과 시 → 문자 기반 청킹 (최후의 수단)
  └─ 호 <= ITEM_THRESHOLD → 호 병합하여 하나의 청크 (항 없는 경우)
      호 초과 → 목 단위로 분리
```

---

## 4. 핵심 규칙: 짧은 항/호 병합

조 전체가 임계값을 넘더라도, **항 또는 호가 여러 개이고 각각이 짧다면 묶어서 하나의 청크**로 저장한다.
불필요하게 쪼개진 청크는 문맥을 해치고 embedding 품질을 낮춘다.

**예시**

```
제53조
① 다음 각 호의 경우에는 주택을 소유한 것으로 본다.  (80자)
  1. 분양권을 가진 경우
  2. 입주권을 가진 경우
② 다음 각 호의 경우에는 주택을 소유하지 아니한 것으로 본다.  (85자)
  1. 상속주택
  2. ...
```

→ ①항(160자) + ②항(170자) = 330자 → `PARAGRAPH_THRESHOLD` 이하이면 **①②를 하나의 청크로 병합**

**병합 판단 기준**

```
현재 누적 길이 + 다음 항/호 길이 <= 임계값  →  병합
초과 시  →  새 청크 시작
```

**주의: 억지로 자르지 않는다**

700~1000자를 넘는 호 단위 청크가 생기더라도, 그것이 원래 하나의 긴 조항이거나
더 이상 쪼갤 의미 단위(목)가 없는 경우라면 글자 수 기준으로 자르지 않는다.
실제 주택공급에 관한 규칙 파싱 결과, 700자 초과 호 청크는 모두 이 경우에 해당했다.

---

## 5. 문맥 단절 방지: 상위 문맥 Prefix

호 또는 목 단위로 청크를 만들 때, **해당 청크만으로는 "무엇에 대한" 설명인지 알 수 없다.**
검색 시 청크가 단독으로 반환되더라도 의미를 파악할 수 있도록 상위 계층의 도입문을 prefix로 포함한다.

### Prefix 포함 규칙

| 청크 단위 | Prefix에 포함할 내용 |
|-----------|----------------------|
| 항 청크   | 조 번호 + 조 제목 |
| 호 청크   | 조 번호 + 조 제목 + 해당 항 도입문 (①②③ 본문) |
| 목 청크   | 조 번호 + 조 제목 + 항 도입문 + 해당 호 본문 |

### 예시

**원문**

```
제53조(주택소유 여부 판정기준)
① 다음 각 호의 경우에는 주택을 소유한 것으로 본다.
  1. 분양권을 가진 경우
  2. 입주권을 가진 경우
```

**나쁜 예 (Prefix 없음)**

```
[청크] 2. 입주권을 가진 경우
```
→ 단독 반환 시 "무엇에 대한" 호인지 알 수 없음.

**올바른 예 (Prefix 포함)**

```
[청크]
제53조(주택소유 여부 판정기준)
① 다음 각 호의 경우에는 주택을 소유한 것으로 본다.
  2. 입주권을 가진 경우
```
→ 단독 반환되어도 "주택소유 판정기준 → 소유로 보는 경우 → 입주권"임을 파악 가능.

## 6. Metadata 설계

```python
metadata = {
    "law":           "주택공급에 관한 규칙",      # 법령명
    "chapter":       "제4장 주택공급 방법",        # 장 이름 (검색 필터용)
    "article":       "53",                        # 조 번호
    "article_title": "주택소유 여부 판정기준",      # 조 제목
    "paragraph":     "1",                         # 항 번호 (①=1, ②=2 ...), 없으면 ""
    "item":          "2",                         # 호 번호, 없으면 ""
    "item_sub":      "",                          # 목 번호 (가=1, 나=2 ...), 없으면 ""
    "chunk_level":   "item",                      # article / paragraph / item / subitem
}
```

**설계 원칙**

- `chapter` — 장 단위 필터 검색에 활용
- `article` — 숫자 문자열로 통일 (`"4-2"` 형식은 `제4조의2`)
- `item` — `"2의3"` → `"2-3"` 으로 정규화 (정렬·필터 가능)
- `paragraph`, `item`, `item_sub` — 없는 경우 `""` (ChromaDB는 `None` 비허용)
- `chunk_level` — 나중에 "조 단위 청크만 검색" 같은 필터에 활용
- 조 제목은 **metadata에도 저장하고 page_content에도 포함** → embedding 품질 보장

---

## 7. 임계값 설정

```python
ARTICLE_THRESHOLD   = 1500   # 조 전체 길이 기준
PARAGRAPH_THRESHOLD = 700    # 항 병합 누적 기준
ITEM_THRESHOLD      = 500    # 호 병합 누적 기준
```

**근거**

한국어 기준 대략 `1글자 ≈ 0.6~0.8 토큰`.

```
512 토큰 ÷ 0.75 ≈ 680자  →  PARAGRAPH_THRESHOLD 700이 안전 하한선
```

> **주의**: 사용하는 embedding 모델의 실제 토큰 한도를 확인한 후 조정할 것.
> `text-embedding-3-small` 등 8191토큰 모델이면 임계값을 훨씬 크게 잡아도 된다.

**파싱 후 분포 확인 방법**

```python
lengths = [len(c.page_content) for c in chunks]
print(f"max: {max(lengths)}, mean: {sum(lengths)/len(lengths):.0f}, min: {min(lengths)}")
```

**실제 주택공급에 관한 규칙 파싱 결과 (참고)**

```
총 청크 수  : 163개
평균 길이   : 566자
최대 길이   : 1371자
chunk_level : article 72개 / item 43개 / paragraph 37개 / subitem 11개

장별 분포
  제1장 총칙                    19개
  제2장 입주자저축               12개
  제3장 입주자모집 및 주택공급 신청  27개
  제4장 주택공급 방법             65개
  제5장 입주자 선정 및 관리        38개
  제6장 보칙                     2개
```

## 8. 전체 요약

| 항목 | 설계 내용 |
|------|-----------|
| 전처리 | 부칙 자동 제거, 개정 이력 제거, 조제목+항기호 줄바꿈 분리 |
| 장(章) | 청킹 단위 아님. 조→장 매핑을 metadata에 포함 |
| 기본 단위 | 조(條) 단위로 파싱 후 길이에 따라 계층 분리 |
| 항 없는 조 | 조 → 호로 바로 분리 (범용 규칙, 제2조 등 포함) |
| 짧은 항/호 병합 | 누적 길이가 임계값 이하면 여러 항/호를 하나의 청크로 묶음 |
| 긴 호 처리 | 목 단위로 분리. 목도 없으면 그대로 유지 (억지로 자르지 않음) |
| 문맥 단절 방지 | 호/목 청크에 상위 항/조 도입문을 prefix로 포함 |
| 조 제목 | metadata + page_content 양쪽에 모두 포함 |
| item 표기 | `"2의3"` → `"2-3"` 으로 정규화 |
| chunk_level | metadata에 청크 단위 명시 (article/paragraph/item/subitem) |
| 임계값 | embedding 모델 토큰 한도 기준으로 설정, 파싱 후 분포 확인하여 조정 |
