# 청약 RAG 시스템 버그 수정 및 품질 개선 리포트

## 1. 개요

이번 세션에서는 두 가지 트랙으로 작업을 진행했다.

1. **버그 수정 트랙**: 신혼부부 특공이 추천 결과에서 누락되고 일반공급 가점이 0으로 표시되는 문제를 추적해 근본 원인을 해결했다.
2. **품질 개선 트랙**: 버그 수정 과정에서 드러난 최종 리포트의 논리적 모순(자격 미달 항목이 1순위로 추천되는 등)을 추가로 발견해 개선했다.

## 2. 버그 수정 트랙

### 2.1 신혼부부 특공 누락 / 일반공급 가점 0점

**원인**: 프론트엔드(`services/payload.py`)가 사용자가 입력한 `child_count_group`, `youngest_child_age_group` 등의 detail 필드를 수집은 하면서도, 백엔드로 보내는 최종 payload(`clean_profile`)를 만드는 과정에서 누락시키고 있었다. 백엔드 스키마(`profile_schema.py`)에도 해당 필드가 정의되어 있지 않아 값이 들어와도 버려졌다.

**조치**:
- `services/payload.py`의 `build_backend_compatible_payload`에 `child_count_group`, `youngest_child_age_group` 필드를 추가
- `app/schemas/profile_schema.py`의 `ProfileInput`에 동일 필드 추가

### 2.2 혼인기간 미확인으로 신혼부부 특공이 부적절하게 후보에 포함됨

**원인**: `node1.py`의 `_is_newlywed_candidate` 함수가 `marital_status == "MARRIED"`이기만 하면 혼인기간을 확인하지 않고 곧바로 신혼부부 특공 후보로 판정했다. 실제로는 혼인 7년 이내가 자격 조건이라, 결혼 20년 차도 후보에 잘못 포함되는 문제가 있었다.

**조치**: `_is_newlywed_candidate`가 `marriage_period`/`marriage_period_years`를 먼저 확인하고, 정보가 전혀 없을 때만 `marital_status`로 폴백하도록 순서를 변경.

### 2.3 일반공급 가점 계산 시 혼인 여부가 잘못 전달됨

**원인**: `node1.py`의 `_build_housing_subscription_score_payload`가 일반공급 가점 계산용 `is_married` 값을 만들 때, "신혼부부 특공 후보 여부를 판단하는 함수"(`_is_newlywed_candidate`, 혼인 7년 이내 여부)를 그대로 재사용했다. 결혼 20년 차는 신혼부부 특공 후보가 아니므로 `False`가 반환되어, 실제로는 기혼자인데 미혼으로 잘못 계산되는 결과를 낳았다.

**조치**: "단순 혼인 여부"만 판단하는 `_is_married` 함수를 신규 추가하고, `is_married` 필드에는 이 함수를 사용하도록 분리.

### 2.4 죽은 코드 정리

- 화면에 렌더링되는 위젯이 없어 항상 빈 값으로만 전달되던 `marriage_period`(카테고리형 혼인기간 옵션)를 `domain/constants.py`, `services/payload.py`, `profile_schema.py`에서 일괄 제거.

### 2.5 가입기간(개월) 계산이 실제 날짜 연산 없이 LLM 추측에 의존하던 문제 (치명적)

**원인**: `src/rag/retriever`를 활용하는 `analyze_subscription_timing`(`rag_tools.py`)이 `bankbook_join_date`를 받았음에도, RAG 검색 쿼리에 가입일 자체를 넣지 않고 실제 날짜 차이도 계산하지 않았다. "가입 기간이 1년 이상인지"를 LLM이 문맥상 추측해서 문장을 생성하는 구조였기 때문에, 가입일이 2024년이고 현재가 2026년인데도 "가입 기간 1년 미만"처럼 사실과 다른 결론이 나올 수 있었다.

**조치**: `_calculate_joined_months` 함수를 신규 추가해 가입일부터 오늘까지의 경과 개월수를 Python으로 직접 계산하고, 이 값을 RAG 쿼리와 system prompt에 "이미 계산된 사실"로 명시해 LLM이 추측하지 않고 그대로 사용하도록 변경. `is_ready` 판정도 계산된 개월수 기준으로 재작성.

## 3. 품질 개선 트랙

### 3.1 명시적 "상세 시뮬레이션 여부" 질문 UI 추가

기존에는 공고문 텍스트 입력 여부로 시뮬레이션 진행 여부를 암묵적으로 추론했으나, 기획 의도(Node 3 분기를 사용자가 명시적으로 선택)에 맞게 예/아니오 카드 선택 UI를 신규 추가했다 (`domain/constants.py`, `views/diagnosis_steps.py`, `services/payload.py`, `views/diagnosis_result.py`).

### 3.2 결과 화면 중복 렌더링 제거

`api_client.py`의 `_normalize_resume_response`가 `report.get("strategy")`(전략 분석 전체 텍스트)를 `guide_message` 폴백으로 사용해, 같은 내용이 화면 상단과 "최종 전략" 섹션에 두 번 출력되던 문제를 수정.

### 3.3 점수 산출 근거를 사용자에게 노출

`node2.py`가 신혼부부/다자녀 특공 계산기의 `score_breakdown`, `matched_items`, `missing_items`, `source_refs`를 버리고 `score`/`max_score`만 추출하던 부분을 확장해 세부 근거를 모두 보존. 일반공급도 `homeless_score`, `dependent_family_score`, `subscription_score`를 모아 별도 `score_breakdown`을 생성. 프론트(`views/diagnosis_result.py`)에 항목별 카드와 한글 라벨 매핑(`SCORE_BREAKDOWN_LABELS`)을 추가해 "왜 이 점수가 나왔는지"를 보여줌. 가독성을 위해 중복도가 높은 "충족 항목" 섹션은 제거.

### 3.4 자격 미달 항목이 1순위로 추천되는 모순 수정

**문제**: 특정 특공이 추첨제(`method == "추첨제"`)이기만 하면 자격 충족 여부와 무관하게 추천 순위 최상단에 올라가는 구조적 결함이 있었다. 사용자 입장에서 "1위 추천인데 신청 자격이 없다"는 모순으로 보였다.

**조치**:
- `strategy_tools.py`의 `_evaluate_lottery_supply`가 `missing_items`/`status`를 반영해 자격 미확정 항목의 우선순위를 낮추도록 수정
- `node2.py`의 `_build_supply_rank`를 "확정 가능 후보"와 "자격 미확정 후보"로 명확히 분리하고, 확정 가능 후보가 하나도 없으면 일반공급을 최우선으로 배치하도록 재작성
- `strategy_tools.py`의 `compare_supply_strategy`에서 1·2순위가 동일 항목으로 중복 추천되던 버그 수정, 일반공급을 secondary 후보 풀에 포함

### 3.5 지역 우선공급 불이익이 당첨 경쟁력 지표에 반영되지 않는 모순 수정

**문제**: "거주지역이 달라 우선공급 대상이 아니다"라는 분석과 "당첨 가능성: 상"이라는 결론이 동시에 존재해 논리적으로 충돌했다. `calculate_winning_probability`가 지역 일치 여부를 입력으로 받지 않는 구조였기 때문이다.

**조치**: `calculate_winning_probability`에 `is_same_region` 파라미터와 페널티 로직(`_get_residency_priority_penalty`)을 추가. `node5.py`에서 `check_regional_priority`를 Agent의 자유 호출이 아닌 고정 순서(STEP 1)로 먼저 실행해 그 결과를 `calculate_winning_probability` 호출 시 명시적으로 전달하도록 구조 변경.

### 3.6 당첨 확률의 단정적 표현을 보수적 표현으로 전환

**문제**: "당첨 가능성: 상"이라는 표현이 실제 경쟁률(신청자 수) 데이터 없이 세대수·점수만으로 산출된 추정치임에도 확정적인 인상을 주었다.

**조치**: `calculate_winning_probability`에 `methodology_notice`(추정 방식과 한계 안내문) 필드를 추가하고, `node5.py` 프롬프트에서 "당첨 가능성" → "참고용 경쟁력 지표"로 표현을 변경. `methodology_notice`를 생략 없이 그대로 노출하도록 규칙 강화.

### 3.7 "신청 자격"과 "1순위 자격" 혼동 해소

**문제**: 결론에서 "자격은 충족했으나 자격은 충족하지 못했다"처럼 같은 단어가 반복되어, 특공/일반공급 신청 자격과 청약통장 1순위 자격이라는 서로 다른 개념이 구분 없이 뒤섞였다.

**조치**: `node5.py` 프롬프트에 두 개념을 명시적으로 구분하는 `[자격 확정 여부]` 섹션과 `[결론 작성 규칙]`을 추가해, "신청은 가능하나 1순위는 미충족"처럼 동사를 다르게 써서 표현하도록 강제.

### 3.8 추천 근거 문구 보강

**문제**: 다른 후보가 없어 일반공급이 선택된 경우 "다른 후보의 자격 확인이 더 필요해 기본 추천 유형을 우선 제시합니다"라는 변명형 문구를 사용해 추천 근거가 약하게 느껴졌다.

**조치**: `compare_supply_strategy`의 해당 분기에서 제외된 후보와 제외 이유(`missing_items`)를 함께 명시하는 적극적 근거형 문구로 변경.

### 3.9 자금 분석에 구체적 행동지침 추가

**문제**: "자금 부담이 높음"이라는 진단에서 그치고 사용자가 다음에 무엇을 해야 하는지 안내가 없었다.

**조치**: `financial.py`의 `analyze_financial_risk`에 리스크 수준별 `action_items`(중도금 대출 확인, 자기자본 확보 방안, 대안 평형 비교 등)를 추가. `node5.py`가 결론 뒤에 "다음 행동" 섹션을 만들어 그대로 노출하도록 프롬프트 규칙 추가.

## 4. 추후 보완점

### 4.1 (우선순위 높음) 실제 경쟁률 데이터 부재

당첨 경쟁력 지표가 신청자 수(실제 경쟁률) 없이 세대수·지역·점수만으로 추정되는 한계가 여전히 남아 있다. 정직한 안내문으로 신뢰도 문제는 완화했으나, 추정의 정확도 자체는 개선되지 않았다. 장기적으로는 청약Home/한국부동산원 등의 과거 분양 단지 실제 경쟁률 데이터를 별도 데이터셋으로 구축해, 유사 조건(지역·평형·공급유형)의 과거 평균 경쟁률을 참조하는 방식으로 고도화가 필요하다. 이는 버그 수정 범위를 넘어선 별도 데이터 파이프라인 구축 작업이다.

### 4.2 "참고용 경쟁력 지표"의 판단 근거 요약 강화

현재는 "중"이라는 등급만 제시되고, 왜 그 등급인지(긍정 요인/부정 요인 구분)가 판단 근거 불릿 목록으로만 나열되어 한눈에 파악하기 어렵다. 긍정 요인과 부정 요인을 구분해 요약하는 섹션을 추가하면 설득력을 높일 수 있다.

### 4.3 자격 미확정 후보의 표현 방식 재검토

현재 "2순위 추천 공급 유형: 생애최초 특공"처럼 자격 미확정 항목도 "추천"이라는 단어를 그대로 사용해, "추천인데 왜 자격이 없지"라는 혼란을 줄 수 있다. "추천 공급 유형"과 "후보 공급 유형(자격 확인 시 검토 가능)"처�럼 표현을 분리하는 것을 검토할 필요가 있다.

### 4.4 최종 결론 상단에 종합 요약 카드 추가

현재 결론은 문단 형태로 흩어져 있어, 신청 가능 여부·1순위 자격·추천 유형·경쟁력·자금 부담을 한 번에 파악하기 어렵다. 보고서 최상단에 핵심 지표를 모은 요약 카드(예: 신청 가능 ○ / 1순위 자격 × / 추천 유형 / 경쟁력 / 자금 부담)를 추가하면 사용자 경험이 개선될 것으로 보인다.

### 4.5 `missing_items`의 의미 명확화 (낮은 우선순위)

`missing_items`라는 이름이 "정보가 없음"과 "정보는 있으나 기준 미달"이라는 두 가지 의미를 함께 담고 있다. 지금은 MVP 단계에서 통장 정보 등 필수 입력값이 항상 채워지는 전제이므로 큰 문제가 되지 않지만, 추후 선택 입력 필드가 늘어날 경우 두 의미를 구분하는 별도 필드 분리를 고려할 필요가 있다. 코드 전반에 이 키 이름이 광범위하게 쓰이고 있어, 변경 시 영향 범위 검토가 필요하다.

### 4.6 RAG 기반 답변(`analyze_subscription_timing`, `check_regional_priority`)의 사실 검증 강화

이번에 가입기간 계산 버그가 발견된 것처럼, RAG 검색 결과를 바탕으로 LLM이 생성하는 답변 중 날짜·횟수 등 정량적 판단이 필요한 부분은 가능한 한 Python으로 직접 계산한 값을 LLM에 "이미 확정된 사실"로 제공하고, LLM에는 제도적 해석과 설명만 맡기는 방향으로 다른 RAG tool들도 점검해볼 필요가 있다.
